## 1. command_worker 동시성

### 현재 상태
| 항목 | 값 | 비고 |
|------|-----|------|
| **Semaphore** | `asyncio.Semaphore(2)` | 최대 동시 브라우저 2개 |
| **폴링 간격** | 10초 | Supabase bot_commands 테이블 폴링 |
| **배치 크기** | 30건 (댓글 게시) | 30건마다 브라우저 재시작 (메모리 보호) |
| **연속 실패 한도** | 5회 | 5회 연속 실패 시 중단 |
| **댓글 간 대기** | 5~15초 | 봇 감지 회피 |
| **블로거 간 대기** | 30~60초 | 봇 감지 회피 |

### 처리 속도 추정
- 블로거 1명 방문 = 게시물 3~5개 × (방문 5초 + AI 2초 + 작성 10초) + 블로거 간 대기 45초 ≈ **2~4분**
- 일일 한도: 10명 × 30댓글 → **약 30~40분/유저**
- Semaphore(2) → **동시 2명 처리 가능**

### 병목
- **10명**: 2명씩 5라운드 = 순차 150~200분 → 허용 시간대(20:00~24:00) 내 처리 **가능하지만 빠듯**
- **50명**: 2명씩 25라운드 = 순차 750~1000분 → **불가능** (4시간 초과)
- **100명**: **물리적으로 불가능**

### 개선 방안
1. **Semaphore 증가** (2→4): WSL2 메모리 충분 시 즉시 적용 가능 (RAM 8GB+ 필요)
2. **시간대 분산**: 유저별 실행 시간 분산 (09:00~23:00)
3. **다중 워커**: 서버 2~3대로 워커 분산 (command_worker에 worker_id 필드 추가)
4. **우선순위 큐**: 유료 유저 우선 처리

## 2. SQLite 멀티유저 동시 접근

### 현재 상태
| 항목 | 값 |
|------|-----|
| **모드** | WAL (Write-Ahead Logging) |
| **busy_timeout** | 10,000ms |
| **재시도** | 3회, 지수 백오프 (0.5s, 1.0s, 1.5s) |
| **DB 분리** | 유저별 파일 (`data/{user_id[:8]}/comments.db`) |

### 강점
- **유저별 DB 분리**: 유저 간 write 경합 없음 (각자 별도 파일)
- WAL 모드: 읽기/쓰기 동시 가능

### 약점
- 같은 유저의 동시 접근 (run + execute 동시 실행 시) → WAL로 대부분 커버되나 극단적 경우 lock
- 백업/마이그레이션 관리가 유저 수에 비례하여 복잡해짐

### 병목 시나리오
- **10명**: 문제 없음 (유저별 분리)
- **50명**: DB 파일 50개 → 파일시스템 부하는 미미, 관리 복잡도 증가
- **100명**: 디스크 I/O 분산 필요. 모니터링 필수

### 개선 방안
1. **현재 유지** (유저 30명까지): 유저별 SQLite가 가장 단순하고 효율적
2. **PostgreSQL 전환** (30명+): Supabase PostgreSQL 직접 사용으로 운영 데이터 통합
3. **Redis 캐시** (50명+): 조회 빈도 높은 데이터(오늘 방문 수, 댓글 수) 캐싱

## 3. Playwright 브라우저 인스턴스 관리

### 현재 상태
- **인스턴스 생성**: 명령 실행마다 새 브라우저 시작 → 종료
- **제한**: Semaphore(2)로 동시 2개까지
- **메모리**: Chromium headless 1개 = 약 200~500MB RAM
- **WSL2 환경**: `--no-sandbox`, `--disable-dev-shm-usage` 플래그 사용

### 메모리 추정
| 유저 수 | 동시 브라우저 | RAM 사용 |
|---------|-------------|---------|
| 2명 | 2개 | 400MB~1GB |
| 4명 | 4개 | 800MB~2GB |
| 8명 | 8개 | 1.6~4GB |

### 개선 방안
1. **브라우저 풀링**: 미리 브라우저를 생성해두고 재사용 (시작 시간 30초 절약)
2. **컨텍스트 분리**: 1개 브라우저에 여러 BrowserContext → 메모리 50% 절약
3. **원격 브라우저**: browserless.io 또는 자체 Chrome 풀 → 로컬 메모리 해방
4. **API 기반 전환** (장기): 네이버 API 사용 가능 시 Playwright 불필요

## 4. Supabase 무료 티어 한계

### 현재 사용량 추정 (1유저 기준)
| 리소스 | 일일 사용량 | 월간 추정 |
|--------|-----------|----------|
| **DB 용량** | ~1MB/일 (run_log + comments) | ~30MB/월 |
| **API 요청** | ~200회/일 (폴링 10초 + CRUD) | ~6,000/월 |
| **Storage** | ~5MB/일 (이미지) | ~150MB/월 |

### Supabase 무료 티어 제한
| 리소스 | 한도 | 유저 N명 시 |
|--------|------|-----------|
| **DB 용량** | 500MB | ~16명 (500/30) |
| **Storage** | 1GB | ~6명 (1000/150) |
| **API 요청** | 월 500,000 | ~83명 (500K/6K) |
| **동시 접속** | 200 | 충분 |
| **Edge Functions** | 500K 호출 | 미사용 |

### 병목 (무료 티어 기준)
- **Storage가 가장 빠르게 소진**: 유저 6명이면 1GB 한도 도달
- DB 용량은 16명까지 여유
- API 요청은 폴링 빈도에 크게 의존

### 개선 방안
1. **이미지 압축 강화**: 현재 max 1920px, JPEG 0.8 → 1280px, 0.6 (50% 절약)
2. **이미지 CDN**: Cloudflare R2 또는 AWS S3 (Storage 분리)
3. **Pro 플랜 전환** ($25/월): DB 8GB + Storage 100GB → 100명까지 여유
4. **폴링 최적화**: 활성 명령 없을 때 30초→60초로 증가 (API 요청 50% 절감)

## 5. 시나리오별 종합 분석

### 유저 10명
| 영역 | 상태 | 조치 |
|------|------|------|
| Worker 동시성 | ⚠️ 빠듯 | 시간대 분산 또는 Semaphore→3 |
| SQLite | ✅ 충분 | 유지 |
| Playwright | ✅ 충분 | RAM 4GB 이상 |
| Supabase | ⚠️ Storage 주의 | 이미지 압축 강화 |
| **총 판정** | **운영 가능** (조건부) | |

| 영역 | 상태 | 조치 |
|------|------|------|
| Worker 동시성 | ❌ 불가 | 다중 워커 + 시간 분산 필수 |
| SQLite | ⚠️ 관리 복잡 | PostgreSQL 전환 검토 |
| Playwright | ❌ 메모리 부족 | 원격 브라우저 또는 서버 증설 |

### 유저 100명
| 영역 | 상태 | 조치 |
|------|------|------|
| Worker 동시성 | ❌ | 분산 워커 클러스터 (3~5대) |
| SQLite | ❌ | PostgreSQL + 커넥션 풀링 |
| Playwright | ❌ | 브라우저 팜 (browserless 등) |

| **총 판정** | **사실상 재설계** | |

## 6. 우선 개선 로드맵

### 즉시 (유저 5명 이내)
- [ ] 폴링 간격 동적 조정 (idle: 60초, active: 10초) — 이미 구현됨 (useBotStatus)
- [ ] 이미지 압축 강화 (1280px, JPEG 0.6)

### 단기 (유저 10~20명)
- [ ] Semaphore 2→4 (서버 메모리 확인 후)
- [ ] 유저별 실행 시간 분산 (bot_settings에 preferred_time 필드)
- [ ] Supabase Pro 플랜 전환
- [ ] 이미지 Storage 분리 (R2/S3)

### 중기 (유저 20~50명)
- [ ] 클라우드 서버 이전 (Oracle Cloud Free 또는 VPS)
- [ ] 다중 워커 지원 (worker_id + 명령 분배)
- [ ] SQLite → PostgreSQL 전환 (운영 데이터 통합)
- [ ] 브라우저 컨텍스트 풀링

### 장기 (유저 50명+)
- [ ] 분산 워커 클러스터 (Kubernetes 또는 서비스 분리)
- [ ] 원격 브라우저 서비스 (browserless.io)
- [ ] 대기열 시스템 도입 (Redis Queue 또는 Celery)
- [ ] 모니터링 인프라 (Prometheus + Grafana)

## 결론

현재 아키텍처는 **유저 5명까지 안정 운영 가능**, **10명까지 조건부 가능**.
가장 먼저 터지는 병목은 **Supabase Storage**(이미지)와 **Worker 동시성**(Semaphore 2).
유저 10명 돌파 전에 Supabase Pro 전환 + 시간 분산 + Semaphore 증가를 적용해야 함.
