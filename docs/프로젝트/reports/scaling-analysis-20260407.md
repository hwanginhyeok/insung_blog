## 1. 처리량 계산 (시나리오별)

### 가정

- "동시 접속자"는 같은 시간대에 봇을 실행하는 사용자 수
- 허용 시간대: 20:00-24:00 (4시간 = 240분)
- 유저 구성 비율: free 60% / basic 25% / pro 15%
- 봇 실행 1회당: free=~5분, basic=~10분, pro=~20분

### 1.1 동시 10명

| 지표 | 값 | 산출 |
|------|-----|------|
| 유저 구성 | free 6, basic 3, pro 1 | 60/25/15% |
| 일일 총 댓글 | 6x10 + 3x50 + 1x400 = **610개** | 각 티어 한도 |
| 일일 총 이웃 방문 | basic 3 + pro 1 = **4명 실행** | |
| 동시 브라우저 필요 | 피크 3-4개 | 10명이지만 시간 분산됨 |
| 봇 처리 총 소요 | ~100분 | 순차 시 10명 x 10분 평균 |
| 세마포어 3 기준 | ~35분 (3병렬) | 100분 / 3 |
| Ollama 부하 | ~4건/분 | 610 / 240분 = 2.5건/분, 피크 x1.5 |
| 4시간 내 완료? | **가능 (여유)** | 35분 << 240분 |

**판정: 현재 시스템으로 문제없이 처리 가능**

### 1.2 동시 50명

| 동시 브라우저 필요 | 피크 10-15개 | 50명 중 20-30%가 동시 실행 |
| 세마포어 3 대비 | **47명 대기** | |
| 봇 처리 총 소요 | ~500분 (순차) | 50명 x 10분 평균 |
| 세마포어 3 기준 | ~167분 (3병렬) | 500분 / 3 |
| 세마포어 5 기준 | ~100분 (5병렬) | 500분 / 5 |
| Ollama 부하 | ~15건/분 | 3,650 / 240분 |
| 4시간 내 완료? | **세마포어 5면 가능, 3이면 빠듯** | |

**문제점**:
1. Elastic Semaphore: max(1, 3//50) = 1슬롯/유저 → 동시 3명만 실행, 나머지 대기열
2. 대기 시간: 마지막 유저는 ~2.5시간 후 실행 시작 가능
3. Ollama 15건/분은 처리 가능하나 여유 없음 (한계 ~40건/분)

**필요 조치**:
- MAX_CONCURRENT_BROWSERS=8-10
- 시간대 분산 (09:00-24:00로 확장)
- 또는 워커 2대 운영

**핵심 병목**:

**필요 아키텍처**:
- 워커 3-5대 분산
- Haiku API 기본 (Ollama 폴백)
- 프록시 풀 도입 (한국 IP 20-30개)
- 시간대 24시간 확장 (해외 사용자 패턴 참고)

## 3. 네이버 감지 리스크 분석

### 3.1 IP당 안전 한도 (추정)

네이버 공식 정책은 비공개이나, 커뮤니티 경험 + 업계 관례 기반 추정:

| 항목 | 안전 범위 | 경계 | 위험 |
|------|---------|------|------|
| IP당 동시 계정 수 | 1-3개 | 4-10개 | 10개+ |
| 계정당 일일 댓글 | 30개 이하 | 30-100개 | 100개+ |
| 계정당 일일 블로그 방문 | 50개 이하 | 50-200개 | 200개+ |
| 댓글 간격 | 10초+ | 5-10초 | 5초 미만 |
| 연속 블로그 방문 | 5개 이하/시간 | 5-20개/시간 | 20개+/시간 |

### 3.2 시나리오별 감지 리스크

| 시나리오 | 동일 IP 계정 수 | 리스크 레벨 | 캡차 예상 빈도 | 계정 제한 위험 |
|---------|--------------|-----------|-------------|-------------|
| 10명 | 3-4 동시 | **낮음** | 거의 없음 | 매우 낮음 |
| 50명 | 10-15 동시 | **중간** | 주 1-2회 | 낮음 |
| 100명 | 20-30 동시 | **높음** | 일 1-5회 | 중간 |
| 500명 | 100-150 동시 | **매우 높음** | 상시 발생 | 높음 |
| 1000명 | 200-300 동시 | **치명적** | 서비스 불가 | 매우 높음 |

> **핵심**: 프록시 없이 50명 이상은 운영 불가능. IP 1개당 계정 3-5개가 안전 한계.

### 3.3 감지 회피 전략 (현재 + 추가)

**현재 구현된 방어:**
- 댓글 간 딜레이: 5-15초 (settings.py:74-75)
- 블로거 간 딜레이: 20-40초 (settings.py:76-77)
- 타이핑 시뮬레이션: 글자당 0.05-0.15초 (settings.py:78-79)
- UA 위장: Chrome 133 (browser.py)
- 자동화 감지 방지: `--disable-blink-features=AutomationControlled`
- 허용 시간대 제한: 20:00-24:00 (자연스러운 활동 시간)
- 오토 블로거 감지/스킵 (점수 70+ 스킵)

**추가 필요 (50명+):**
- IP 로테이션 (프록시 풀)
- UA 랜덤화 (버전별 가중치)
- 유저별 딜레이 패턴 랜덤화 (현재 고정 범위 → 유저마다 다른 범위)
- 브라우저 핑거프린트 다양화 (뷰포트, 언어, 타임존)
- 캡차 자동 해결 (2Captcha 연동) 또는 캡차 시 자동 대기

**추가 필요 (100명+):**
- 유저별 전용 IP 할당
- 네이버 rate-limit 패턴 학습 + 동적 딜레이 조정
- 계정 건강도 모니터링 (캡차 빈도, 댓글 삭제율 추적)
- 실패율 기반 자동 쓰로틀링

**추가 필요 (500명+):**
- 모바일 프록시 혼합 (감지율 최저)
- 리전별 IP 분산 (서울, 경기, 부산 등)
- 봇 활동 패턴 AI화 (사람 행동 모사 ML 모델)
