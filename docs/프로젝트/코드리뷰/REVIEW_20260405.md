### 4. 065ea45 — perf(ollama): 댓글 생성 Ollama 병렬 처리

**변경 범위**: ai_comment.py

**내용**: 순차 Ollama 호출을 `ThreadPoolExecutor(max_workers=3)`로 병렬화. 배치 생성과 폴백 모두 적용.

| 항목 | 판정 | 근거 |
|------|------|------|
| 아키텍처 | **PASS** | ThreadPoolExecutor는 I/O 바운드(HTTP 요청)에 적합. `as_completed`로 결과 순서 무관하게 처리. 인덱스 기반으로 결과를 올바른 위치에 배치. |
| 안전성 | **CONCERN** | `future.result()`에 timeout이 없음. Ollama가 행(hang)하면 쓰레드가 무한 대기. `_generate_one` 내부에서 예외가 발생하면 `future.result()`에서 re-raise되는데, try-except가 없어서 하나의 실패가 전체 배치를 죽일 수 있음. |
| 성능 | **PASS** | 3개 병렬 워커는 Ollama의 `OLLAMA_NUM_PARALLEL=3`과 매칭. 환경변수로 설정 가능(`OLLAMA_PARALLEL`). |
| 테스트 | **ISSUE** | 병렬화 코드에 대한 테스트 없음. 특히 에러 케이스(일부 쓰레드 실패, 타임아웃)에 대한 검증 필요. |
| 코드 품질 | **CONCERN** | `_generate_one`과 `_fallback_one`이 거의 동일한 코드. 클로저로 함수 내부에 정의되어 있어 리팩토링 어려움. 하나의 공통 함수로 추출 가능. |

**주요 발견 사항**:
- `_generate_one`/`_fallback_one` 코드 중복이 심함. 동일한 로직을 두 번 정의하고 있어 하나를 수정하면 다른 것도 수정해야 함.
- `recent_comments_local.append(comment)`가 병렬 쓰레드에서 호출되는데, Python 리스트의 `append`는 GIL로 인해 thread-safe하긴 하지만, 중복 검사(`_is_similar`)가 배치 내에서는 race condition 가능 — 동시 생성된 댓글이 서로의 중복을 감지하지 못할 수 있음.
- 병렬 실행 중 하나의 future에서 예외 발생 시, `as_completed` 루프에서 처리되지 않으면 나머지 결과가 무시됨.
