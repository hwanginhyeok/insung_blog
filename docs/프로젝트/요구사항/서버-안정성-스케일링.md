## 발견된 버그 및 수정 (2026-04-05)

### BUG-1: Elastic Semaphore 미작동 — ✅ 수정
- **문제**: `acquire_user_slot()`/`release_user_slot()`이 정의만 되고 호출 안 됨
- **영향**: 유저 A가 슬롯 전체 독점, 유저 B 무한 대기
- **수정**: `process_command()`에서 acquire/release 호출 추가
- **커밋**: `cd06a90`

### BUG-2: Task 생성 루프 무한 스핀 — ✅ 수정
- **문제**: `main_loop()`에서 `create_task()` 후 `continue` → 슬롯 만석인데도 계속 Task 생성
- **영향**: CPU 100% 스핀, 메모리 낭비
- **수정**: `active_tasks >= MAX`이면 claim 스킵하고 1초 대기
- **커밋**: `cd06a90`

### BUG-3: `/comment/execute` API 30분 블로킹 — ✅ 수정
- **문제**: HTTP 핸들러 안에서 Playwright로 댓글 200개 순차 실행
- **영향**: API 서버 이벤트 루프 전체 점유, 다른 요청 처리 불가
- **수정**: bot_commands 큐에 execute 명령 삽입 → 워커가 비동기 처리 → API 즉시 응답
- **커밋**: `65568be`

### BUG-4: `/publish` API 브라우저 블로킹 — ✅ 수정
- **문제**: HTTP 핸들러에서 Playwright 브라우저 직접 실행
- **영향**: 발행 중 API 서버 블로킹
- **수정**: bot_commands 큐에 publish 명령 삽입 → 워커가 비동기 처리
- **커밋**: `65568be`

### BUG-5: auto_reply 브라우저 args 누락 — ✅ 수정
- **문제**: `handle_auto_reply`에서 `p.chromium.launch(headless=True)` 직접 호출
- **영향**: `--no-sandbox`, `--disable-dev-shm-usage` 미적용 → WSL2 불안정
- **수정**: `_BROWSER_ARGS` 적용
- **커밋**: `cd06a90`

### 세마포어 구조 개선 — ✅ 수정
- **문제**: `handle_run()`이 전체 `run()` 실행 동안 세마포어 점유 (블로거 간 30-60초 대기 포함)
- **영향**: 대기 중에도 슬롯 잠금 → 다른 유저 불필요한 대기
- **수정**: orchestrator 내부에서 브라우저 사용 구간에만 세마포어 acquire/release, delay 중에는 슬롯 반환
- **커밋**: `6f67964`

## 스케일 임계점 분석

### 1-5명 (현재)
- **상태**: 정상 운영
- **병목**: 없음

### 10명
- **1차 병목**: 시간 창 부족 — 봇 실행 시간대(20-24시) 4시간에 ~20회만 처리 가능
- **2차 병목**: Supabase Storage (이미지 발행 시 1GB 한도)
- **대응**: 시간대 확장, MAX_CONCURRENT_BROWSERS=5 (완료)

### 50명
- **1차 병목**: 워커 처리량 — 50명 중 30명은 다음날로 밀림
- **2차 병목**: Supabase Free tier API 요청 한도 (500K/월) 근접
- **대응 필요**:
  - Supabase Pro 전환 ($25/월)
  - 멀티프로세스 워커 (Celery + Redis)

### 100명
- **1차 병목**: 세마포어 + 단일 IP에서 100계정 운영 → 네이버 IP 차단 위험
- **2차 병목**: Supabase API 쿼터 초과 (600K > 500K)
- **대응 필요**:
  - 분산 워커 (다중 서버)
  - 프록시 풀 (IP 분산)
  - Supabase Pro
  - Task 생성 루프 추가 최적화

### 1000명
- **상태**: 현재 아키텍처로 불가
- **대응 필요**:
  - Celery + Redis 분산 큐
  - 멀티서버 (각 서버 브라우저 10개)
  - 주거용 프록시 풀
  - Ollama → Claude API 전환 (처리량)
  - uvicorn 멀티워커

## 처리량 계산

### 유저 1명의 run 소요 시간
- 블로거 10명 방문 × (방문 3분 + 대기 45초) ≈ 37분

### 시간당 처리량 (세마포어 개선 후)
- 슬롯 5개, 대기 중 슬롯 반환
- 이론상: 슬롯당 시간당 ~1.6회 = 5 × 1.6 = **시간당 8회**
- 4시간 창: **~32회/일**

### Ollama 댓글 생성 처리량
- 3 병렬 워커 (OLLAMA_PARALLEL=3)
- ~40 댓글/분 = 2,400 댓글/시간
- 50명 × 30댓글 = 1,500댓글/일 → 충분
