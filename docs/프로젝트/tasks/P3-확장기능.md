## EXT-04: 다중 페르소나

### 배경

W5에서 `user_personas.user_id`에 UNIQUE 제약이 걸려 있어 1 user : 1 persona만 가능했음.
블로거가 여러 스타일(맛집용, 여행용 등)을 운영하는 현실에 맞게 1:N으로 전환.

### DB 마이그레이션 (`00012_enable_multiple_personas.sql`)

```sql
-- 1:1 제약 해제
ALTER TABLE user_personas DROP CONSTRAINT IF EXISTS user_personas_user_id_key;

-- 기본 페르소나 플래그
ALTER TABLE user_personas ADD COLUMN IF NOT EXISTS is_default BOOLEAN DEFAULT false;

-- 기존 데이터: 첫 페르소나를 기본으로 설정
UPDATE user_personas SET is_default = true
WHERE id IN (
  SELECT DISTINCT ON (user_id) id FROM user_personas ORDER BY user_id, created_at ASC
);

-- 사용자당 기본 페르소나 1개만 보장 (부분 유니크 인덱스)
CREATE UNIQUE INDEX idx_user_personas_default
  ON user_personas(user_id) WHERE is_default = true;
```

### 핵심 변경: `.single()` → `.maybeSingle()` 마이그레이션

1:1에서 1:N으로 전환하면서 `.single()`을 사용하던 모든 쿼리를 `.maybeSingle()`로 변경:

| 파일 | 변경 |
|------|------|
| `lib/ai/generate-post.ts` | `loadUserPersona(userId, personaId?)` — personaId 파라미터 추가, is_default 쿼리 분기 |
| `app/api/persona/crawl/route.ts` | personaId로 재크롤링 지원, 첫 페르소나 자동 is_default |
| `app/api/persona/feedback/route.ts` | GET/POST 모두 personaId 기반 조회 |
| `app/api/generate/route.ts` | body에 personaId 추가 → generatePost() 전달 |
| `app/api/regenerate/route.ts` | body에 personaId 추가 → regeneratePost() 전달 |

### 신규 파일

| 파일 | 역할 |
|------|------|
| `app/api/persona/list/route.ts` | GET: 사용자 페르소나 목록 (created_at 순) |
| `app/api/persona/default/route.ts` | POST: 기본 페르소나 지정 (admin 클라이언트로 cross-row 업데이트) |
| `app/(dashboard)/persona/[id]/page.tsx` | 개별 페르소나 상세 (기존 persona/page.tsx 로직 이동, 이름 편집/삭제 추가) |

### UI 변경

**persona/page.tsx (전면 개편)**:
- 기존: 단일 페르소나 상세 뷰 (아코디언 7카테고리)
- 변경: 그리드 카드 목록 + "새 페르소나 추가" 폼 + 기본 설정 버튼
- 카드 클릭 → `/persona/[id]` 상세 페이지로 이동

**write/page.tsx**:

- 2개 이상일 때 드롭다운 선택기 표시, 기본 페르소나 자동 선택

## EXT-02: 콘텐츠 캘린더

### DB 마이그레이션 (`00013_create_content_calendar.sql`)

```sql
CREATE TABLE content_calendar (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  planned_date DATE NOT NULL,
  topic TEXT NOT NULL,
  category TEXT CHECK (category IN ('맛집','카페','여행','일상','기타')),
  memo TEXT,
  status TEXT DEFAULT 'planned'
    CHECK (status IN ('planned','in_progress','published','cancelled')),
  generation_id UUID REFERENCES generation_queue(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 인덱스
CREATE INDEX idx_content_calendar_user_date ON content_calendar(user_id, planned_date);

-- RLS (4개 정책: SELECT, INSERT, UPDATE, DELETE)
```

### API (`app/api/calendar/route.ts`)

| Method | 기능 | Body/Query |
|--------|------|-----------|
| GET | 월별 이벤트 + 게시물 조회 | `?year=2026&month=3` |
| POST | 이벤트 생성 | `{ planned_date, topic, category?, memo? }` |
| PATCH | 이벤트 수정 | `{ id, topic?, status?, ... }` |
| DELETE | 이벤트 삭제 | `?id=xxx` |

### UI (`app/(dashboard)/calendar/page.tsx`)

순수 React + HTML 테이블로 구현 (외부 캘린더 라이브러리 없음):

## EXT-03: 성과 분석

### DB 마이그레이션 (`00014_create_post_analytics.sql`)

```sql
CREATE TABLE post_analytics (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  post_url TEXT NOT NULL,
  post_title TEXT,
  view_count INTEGER DEFAULT 0,
  comment_count INTEGER DEFAULT 0,
  like_count INTEGER DEFAULT 0,
  crawled_at TIMESTAMPTZ DEFAULT NOW(),
  generation_id UUID REFERENCES generation_queue(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 최신 스냅샷 뷰 (URL별 최신 1건)
CREATE OR REPLACE VIEW user_post_stats AS
SELECT DISTINCT ON (user_id, post_url)
  id, user_id, post_url, post_title,
  view_count, comment_count, like_count,
  crawled_at, generation_id
FROM post_analytics
ORDER BY user_id, post_url, crawled_at DESC;
```

**설계 포인트**: 크롤링마다 행을 INSERT하여 시계열 데이터 축적.
`DISTINCT ON`으로 URL별 최신 스냅샷만 추출하는 뷰 제공.

## EXT-05: OAuth 소셜 로그인 (Kakao + Naver)

### 구현 방식

카카오/네이버는 Supabase 내장 provider가 아니므로 직접 OAuth 플로우 구현:

```
사용자 → /api/auth/kakao/login (state 쿠키 저장 + 리다이렉트)
     → 카카오 인증 페이지
     → /api/auth/kakao/callback (토큰 교환 + 사용자 매칭 + 세션 생성)
     → /dashboard
```

### 세션 생성 전략: Magic Link + verifyOtp

```typescript
// 1. admin API로 magic link 생성 (이메일 발송 안 됨)
const { data: linkData } = await admin.auth.admin.generateLink({
  type: "magiclink",
  email: userEmail,
});

// 2. 서버 측에서 즉시 OTP 검증 → 세션 쿠키 설정
const { error } = await supabase.auth.verifyOtp({
  token_hash: linkData.properties.hashed_token,
  type: "magiclink",
});
```

기존 Supabase Auth 세션과 완벽 호환 — middleware의 `getUser()` 변경 불필요.

### DB 마이그레이션 (`00015_add_oauth_providers.sql`)

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS kakao_id TEXT;
ALTER TABLE users ADD COLUMN IF NOT EXISTS naver_id TEXT;

-- 부분 유니크 인덱스 (NULL 제외)
CREATE UNIQUE INDEX IF NOT EXISTS idx_users_kakao_id
  ON users(kakao_id) WHERE kakao_id IS NOT NULL;
CREATE UNIQUE INDEX IF NOT EXISTS idx_users_naver_id
  ON users(naver_id) WHERE naver_id IS NOT NULL;
```

### 계정 연결 정책

```
OAuth 로그인 시:
1. kakao_id/naver_id로 기존 사용자 검색 → 있으면 바로 로그인
2. 없으면 이메일로 검색 → 있으면 OAuth ID 연결 + 로그인
3. 둘 다 없으면 → 새 auth.users 생성 (트리거로 public.users 자동 생성) → OAuth ID 연결
4. 이메일 없는 OAuth 사용자 → `kakao_{id}@oauth.local` 형식 임시 이메일
```

### CSRF 방지

- `crypto.randomUUID()`로 state 생성 → `oauth_state` httpOnly 쿠키 저장
- 콜백에서 URL 파라미터의 state와 쿠키의 state 대조 검증
- 불일치 시 `/login?error=...`로 리다이렉트

### 신규 파일

| 파일 | 역할 |
|------|------|
| `app/api/auth/kakao/login/route.ts` | 카카오 인증 URL 리다이렉트 |
| `app/api/auth/kakao/callback/route.ts` | 콜백: 토큰 교환 + 사용자 매칭 + 세션 |
| `app/api/auth/naver/login/route.ts` | 네이버 인증 URL 리다이렉트 |
| `app/api/auth/naver/callback/route.ts` | 콜백: 토큰 교환 + 사용자 매칭 + 세션 |

### 로그인 페이지 변경 (`login/page.tsx`)

- "또는" 구분선 추가
- 카카오 버튼 (#FEE500 노랑)
- 네이버 버튼 (#03C75A 초록)
- 기존 redirect 파라미터를 OAuth 플로우에도 전달

