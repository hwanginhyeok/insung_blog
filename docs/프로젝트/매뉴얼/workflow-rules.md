# Workflow Rules

> CLAUDE.md에서 분리된 상세 워크플로우 규칙. 코드 수정 후 참조.

## 작업 순서 (코드 작업 시)

1. 현재 코드 읽기 (Read) → 변경 범위 파악
2. 수정 (Edit/Write) → 최소 변경 원칙
3. 구문 검증 (`py_compile`)
4. **코드리뷰 필수** — 아래 프로세스 수행
5. 리뷰 결과를 사용자와 논의 → Critical/Major 이슈 수정
6. CODE_MAP.md 갱신 (파일 추가/삭제 시)
7. TASK.md 갱신

---

## 코드리뷰 프로세스 (필수)

> 코드 수정 후 반드시 리뷰를 수행한다. 리뷰 없는 코드는 커밋하지 않는다.

### 리뷰 체크리스트

| 카테고리 | 체크 항목 |
|----------|-----------|
| **보안** | API 키 하드코딩 없는지, 사용자 입력 검증, .env 노출 없는지 |
| **안정성** | 폴백 메커니즘 존재, 에러 처리 적절한지, 타임아웃 설정 |
| **컨벤션** | coding-rules.md 준수 (네이밍, 상수 중앙화, 로깅 패턴) |
| **셀렉터** | 네이버 DOM 셀렉터가 폴백 체인으로 구성되었는지 |
| **타입** | type hints 누락 없는지, return type 명시 |
| **중복** | 기존 유틸/함수 재사용 가능한 부분이 있는지 |

### 심각도 분류

| 심각도 | 기준 | 조치 |
|--------|------|------|
| **Critical** | 보안 취약점, 데이터 손실 위험, 크래시 | 반드시 수정 |
| **Major** | 로직 오류, 컨벤션 위반, 폴백 누락 | 수정 권장 |
| **Minor** | 코드 스타일, 변수명 개선, 불필요한 import | 사용자 판단 |
| **Suggestion** | 성능 개선, 구조 개선 아이디어 | 참고용 |

### 리뷰 흐름 (브리핑 우선 원칙)

> **절대 규칙**: 이슈를 발견하면 **먼저 사용자에게 브리핑**한다.
> 혼자 판단해서 바로 수정하지 않는다. 사용자가 문제를 이해하고 함께 결정한다.

```
코드 수정 완료
    ↓
리뷰 수행 (체크리스트 기반)
    ↓
★ 사용자 브리핑 (이슈 수정 전) ★
    │  1. 이슈 목록 (심각도 + 파일 + 설명)
    │  2. 각 이슈가 왜 문제인지 설명
    │  3. 수정 방안 제안 (대안이 있으면 함께 제시)
    ↓
사용자와 논의 → 수정 방향 합의
    ↓
합의된 방향으로 수정 → 재검증
    ↓
REVIEW_LOG.md 기록
    ↓
커밋 가능
```

### 브리핑 형식

```
## 코드리뷰 브리핑

### 요약: Critical N / Major N / Minor N

### 이슈 목록
| # | 심각도 | 파일:줄 | 문제 | 왜 문제인지 | 수정 제안 |
|---|--------|---------|------|-------------|-----------|
| 1 | Major  | file.py:42 | ... | ... | ... |

### 결정 필요
- 이슈 #N: A안 vs B안 — 어떻게 할까요?
```

## 코드리뷰 기록 (REVIEW_LOG.md)

- **파일**: `docs/프로젝트/코드리뷰/REVIEW_LOG.md`
- **기록 시점**: 리뷰 결과를 사용자와 논의한 직후, 커밋 전
- **최신순** 기록 (새 항목을 위에 추가)

### 기록 템플릿

```markdown
## YYYY-MM-DD — [작업 요약]

- **TASK**: #번호
- **변경 파일**: file1.py, file2.py
- **이슈 요약**: Critical 0 / Major 1 / Minor 2
- **조치**: [수정 내역 또는 "이슈 없음"]
- **판정**: 머지 가능 / 수정 후 머지
```

## TASK.md 실시간 갱신 규칙

| 트리거 | 행동 |
|--------|------|
| **작업 착수 시** | "현재 진행 중"에 등록 + 상태 → `진행` |
| **작업 완료 시** | 완료 섹션으로 이동 + "현재 진행 중" 비우기 |
| **새 작업 발견 시** | 즉시 작업 현황에 등록 |
| **커밋 직전** | TASK.md + CODE_MAP.md + REVIEW_LOG.md 모두 최신인지 점검 |
| **세션 종료 시** | 미완료 작업 진행 상황 기록 |


## 승인 워크플로우 개발 규칙 (2026-03-04 추가)

> 텔레그램 봇 댓글 승인/실행 기능 개발 시 준수 사항

### 데이터 흐름

```
[수집/AI 생성] → DB(pending) → 텔레그램(/pending) → 사용자 승인 → DB(queued) → /execute → 실제 실행 → DB(posted)
```

### 구현 체크리스트

| 단계 | 체크 항목 | 파일 |
|------|----------|------|
| **저장** | `pending_comments` 테이블에 status='pending' 저장 | `database.py` |
| **조회** | `/pending` 명령어로 pending 상태 조회 | `telegram_bot_simple.py` |
| **승인** | status='approved' 또는 별도 실행 큐에 추가 | `telegram_bot_simple.py` |
| **거부** | status='rejected' 변경 + 사용자 알림 | `telegram_bot_simple.py` |
| **실행** | `/execute`로 승인된 댓글 일괄 처리 | `telegram_bot_simple.py` |
| **결과** | 실행 결과 텔레그램 통보 + DB 업데이트 | `telegram_bot_simple.py` |

### UI/UX 규칙

1. **즉각적인 피드백**: 버튼 클릭 시 즉시 "승인됨" / "거부됨" 메시지
2. **배치 처리**: 승인은 즉시, 실제 실행은 `/execute`로 일괄
3. **상태 표시**: 현재 승인 개수 / 전체 개수 표시 (예: "3개 승인됨 / 5개 대기")
4. **오류 처리**: 실행 실패 시 재시도 큐에 자동 추가

### 데이터베이스 상태 흐름

```
pending → approved → posted (성공)
   ↓         ↓
rejected   retry_queue (실패 시)
```

### 테스트 시퀀스

1. 테스트 데이터 생성: `scripts/add_test_comments.py`
2. `/pending` 조회 → 3개 표시 확인
3. `✅ 승인` 클릭 → "승인됨" 메시지 + 카운트 증가
4. `❌ 거부` 클릭 → "거부됨" 메시지
5. `/execute` → 일괄 실행 (dry-run 모드로 테스트)
6. 결과 통보 메시지 확인
