### 8. DUPLICATE-FIX
중복 댓글 근본 원인 수정. 분석 리포트: `docs/프로젝트/briefings/DUPLICATE-COMMENTS_root_cause_20260329.md`

**배경**: `pending_comments`에 154개 중복 쌍(약 420건) 존재. 실제로 같은 블로그 글에 중복 댓글 게시됨.
근본 원인: ① Supabase UNIQUE 제약 없음 ② `is_post_commented()`가 SQLite만 체크 ③ 레이스 컨디션 ④ 게시 중 중복 제거 없음.

**작업 순서**:

#### 8-1. 기존 중복 데이터 정리 (dry-run 먼저)
`tools/fix_duplicate_comments.py` 스크립트 작성:
- Supabase `pending_comments`에서 `(post_url, user_id)` 중복 조회
- 각 그룹에서 우선순위(posted > approved > pending > rejected > failed)로 1개만 유지
- 나머지를 `status='rejected', fail_reason='중복 자동 제거'`로 업데이트 (DELETE 아님)
- dry-run 모드: `--dry-run` 플래그로 실제 변경 없이 대상만 출력
- dry-run 결과 확인 후 실제 실행

#### 8-2. `add_pending_comment_sb()` 중복 사전 체크
파일: `src/storage/supabase_client.py`
- INSERT 전에 `(post_url, user_id)` + `status IN ('pending', 'approved')` 조회
- 있으면 기존 id 반환 (재삽입 안 함)
- 없으면 기존 로직대로 INSERT

#### 8-3. `is_post_commented()` Supabase 동시 체크
파일: `src/storage/database.py`
- SQLite 체크 후 False면 Supabase도 체크
- Supabase `pending_comments`에서 `status IN ('pending', 'approved', 'posted')` 조회
- 있으면 True 반환 (중복 방지)
- 실패(네트워크 오류 등)하면 경고 로그 후 SQLite 결과만 사용

#### 8-4. `command_worker.py` 게시 전 중복 제거
파일: `command_worker.py`
- `approved` 목록에서 `post_url` 기준 중복 제거 (set으로 seen 추적)
- 중복 항목은 즉시 `rejected`로 상태 변경 + 경고 로그

#### 8-5. Supabase 마이그레이션 SQL 파일 생성 (실행은 수동)
파일: `supabase/migrations/20260329_add_unique_pending_comments.sql`
- 중복 제거 후 `UNIQUE(post_url, user_id)` 제약 추가 SQL 작성
- 실제 실행은 하지 말 것 (스키마 변경은 수동 진행)

#### 8-6. 테스트 실행
- `pytest tests/test_duplicate_prevention.py -v`
- 구문 검증: 수정한 모든 .py 파일

브리핑 파일: `docs/프로젝트/briefings/DUPLICATE-FIX_briefing.md`

- 파일 삭제(rm) 금지

- 실행 중인 tmux 세션(blog) 건드리지 않기
- 새 패키지 설치(pip install, npm install) 금지

### 허용 사항
- 파일 읽기, 생성, 수정 (기존 파일 편집 포함)
- git add, git commit (커밋 메시지 한국어, Conventional Commits)
- python 스크립트 실행 (테스트, 구문 검증)
- 로그 파일 생성/작성
- TASK.md 갱신 (작업 상태 업데이트)
- CODE_MAP.md 갱신

### 작업 방식
1. TASK.md 읽고 작업 대상 파악
2. 작업 착수 시 TASK.md 상태를 '진행'으로 갱신
3. 코드 수정 전 반드시 CLAUDE.md, docs/프로젝트/매뉴얼/coding-rules.md 읽고 준수
4. 코드 수정 후 반드시 구문 검증: python -c "import py_compile; py_compile.compile('파일.py', doraise=True)"
5. 웹(Next.js) 코드 수정 시 apps/ 디렉토리에서 작업, TypeScript 타입 안전성 유지
6. 작업 완료 시 TASK.md 상태를 '완료'로 갱신
7. 각 작업 완료 시 git commit (커밋 메시지 한국어, Conventional Commits)
8. 에러 발생 시 3번까지 재시도, 실패하면 TASK.md에 에러 내용 기록하고 다음 작업으로

### ⚠️ 서비스 상태 확인 (필수 — 매 작업 후)
`command_worker.py`, `src/orchestrator.py`, `src/commenter/` 등 봇 핵심 파일 수정 시:
```bash
systemctl --user status blog-api blog-worker blog-telegram | grep -E "●|Active:"
```
- 3개 모두 `active (running)` → 정상
- 하나라도 `inactive/failed` → 즉시 재시작:
  ```bash
  systemctl --user restart blog-api blog-worker blog-telegram
  sleep 2
  systemctl --user status blog-api blog-worker blog-telegram | grep -E "●|Active:"
  ```
- 재시작 후에도 failed → 로그 확인:
  ```bash
  journalctl --user -u blog-worker -n 30 --no-pager
  ```
- 워커 중복 프로세스 반드시 1개 확인:
  ```bash
  ps aux | grep command_worker | grep -v grep | wc -l  # 반드시 1
  ```
  2 이상이면: `systemctl --user stop blog-worker && pkill -f command_worker.py 2>/dev/null && sleep 2 && systemctl --user start blog-worker`

**모든 작업 완료 후 최종 서비스 상태를 overnight_report.md에 기록할 것.**
